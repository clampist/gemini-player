## AIテストのジャーニー

### ツール選定の背景
- まず `ChatGPT` を評価。安定した `data-testid` 属性が存在し、Cookie ログインで操作範囲を拡張できたため。ただし認証後しばらくして Playwright 由来のアクセスと判定され、検証ステップを突破できなくなった。
- 認証不要な代替案を探し、`Perplexity.ai` と `Gemini` を試行。ところが Perplexity 側も自動化を検知し、チャット応答を返さなくなった。
- 最終的に `Gemini` をターゲットとした。公開ページは未ログインでアクセスでき、Playwright による基本的な操作も一通り成立したため。

### 開発アプローチ
- まず MVP を成立させることを優先。ローカルでの end-to-end テスト実行から GitHub 連携、Playwright HTML レポートの公開までを一気通貫で検証。
- 垂直スライスが通った後に段階的に拡張し、ローカルと CI で同一ワークフローを再現できるよう整備。

### テスト設計のポイント
- 未ログインシナリオ向けの軽量構成を意識し、`data` や `support` などシンプルなディレクトリ構造を採用。
- 設定は専用ファイルに切り出し、業務ロジックとテストコードを分離。セレクタを集中管理してドリフトを抑制。

### 運用上の注意
- Gemini は自動化アクセスを一定頻度で制限または遮断する。ブラウザは Chromium のみに絞り、実行間隔を空けることを推奨。
- 公開 AI サイトのセレクタは予告なく変わる可能性があるため、定期的なロケータの棚卸しが必要。
